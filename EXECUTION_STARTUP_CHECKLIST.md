## 6. 执行期间日常流程

### 6.1 每日站会 (Daily Standup)

**时间**: 每日 10:00-10:15 (15 分钟)  
**参与**: 所有 Team Leads + 1-2 代表  
**形式**: 在线会议 (Zoom) + Slack 记录

**议程** (每人 3 分钟):
1. 昨天完成了什么？(代码行数、测试通过情况)
2. 今天计划做什么？(关键任务)
3. 有什么阻挡吗？(问题、依赖、风险)

**会议记录**: 发送到 #daily-standup

### 6.2 每周技术评审 (Weekly Technical Review)

**时间**: 每周五 14:00-16:00 (120 分钟)  
**参与**: PM + 所有 Team Leads + Architects  
**地点**: 大会议室 / 在线

**议程**:
```
14:00-14:10  周度总结 (代码量、测试、进度)
14:10-14:30  PTP Team 进展展示
14:30-14:50  TSN Team 进展展示
14:50-15:10  硬件 Team 进展展示
15:10-15:30  跨团队集成进展
15:30-15:50  风险评估 & 问题解决
15:50-16:00  下周计划确认
```

**产出**: 
- 周度进度报告
- 风险寄存器更新
- 决策记录
- 下周目标确认

### 6.3 代码审查 (Code Review)

**流程**:
```
Developer 完成功能 → 创建 Pull Request
           ↓
Code Review (Lead + 1 peer) → 反馈
           ↓
修改 & 再次审查 → 批准
           ↓
Merge to develop → CI/CD 自动测试
           ↓
测试通过 → 关闭 PR
```

**要求**:
- 每个 PR 必须有 2 个 Approve
- 测试覆盖率 >= 80%
- 0 编译警告
- Code style 符合规范

## 7. 执行期间常见问题 (FAQ)

### Q1: 如果某个团队进度落后怎么办？
**A**: 
1. 每日站会立即识别问题
2. PM 分析根本原因
3. 调整资源分配或任务优先级
4. 启动风险缓解措施
5. 必要时进行跨团队支援

### Q2: 代码实现中发现设计有问题？
**A**:
1. 立即报告给相应架构师
2. 召开快速设计评审会
3. 决定是修改设计还是修改实现
4. 更新相关文档
5. 记录在决策日志中

### Q3: 硬件设计遇到成本问题？
**A**:
1. 汇总到硬件 Lead
2. 评估替代器件 / 设计方案
3. 成本对标 BOM 目标 (¥361)
4. 必要时启动采购谈判
5. 更新 BOM 与成本分析

### Q4: 如何处理紧急 Bug？
**A**:
1. 立即分类为 P0/P1/P2/P3
2. P0/P1 停止当前任务立即修复
3. 记录在问题追踪系统
4. 进行原因分析与改进
5. 防止类似问题再次发生

### Q5: 如何平衡代码质量与时间压力？
**A**:
1. 坚守单元测试 80%+ 要求
2. 代码审查不降低标准
3. 将性能优化推到 M2.2
4. 优先完成功能，然后迭代优化
5. 定期回顾与改进流程
